# Chat Moderation, Retention, and Tab Notifications Design

**Date:** 2026-07-01

**Status:** Approved

## Goal

Protect PingPing chat from rapid spam, cap stored history at 36 messages per room, remove files that no longer belong to retained messages, and surface unread activity in the browser tab title.

## Scope

This design adds:

- a global per-member rate limit of 18 user messages in any rolling 60-second window;
- a five-minute mute after an attempted nineteenth message;
- a temporary `Súc vật <Tên>` display name during the mute;
- one system message in the room where the violation occurred;
- a hard retention limit of 36 messages per room;
- reliable cleanup of attachments belonging to removed or rejected messages;
- an unread count in the document title; and
- browser notifications only while the PingPing page remains open but hidden.

Push notifications after the page has been closed are explicitly out of scope.

## Architecture

All user-message writes go through one security-definer Postgres RPC, `send_chat_message`. Direct anonymous inserts into `messages` are revoked. The RPC authenticates the supplied member ID and login code, serializes concurrent sends for that member, enforces the rate limit and mute, inserts accepted or system messages, and prunes the target room in one database transaction.

The existing vanilla JavaScript controller calls the RPC instead of inserting into `messages`. Supabase Realtime remains responsible for delivering new messages and member-state changes to connected clients.

Storage deletion is decoupled from the message transaction. Paths requiring deletion are recorded durably in a cleanup queue. A cleanup worker removes each object through the Supabase Storage API and deletes the corresponding attachment metadata only after object deletion succeeds. Failed work remains queued for retry.

## Data Model

### Members

Add `chat_muted_until timestamptz`. The canonical `name` remains unchanged.

An effective display-name helper returns `Súc vật <name>` while `chat_muted_until > now()` and returns the canonical name otherwise. Clients also schedule a local refresh at the mute expiry so the visible name and composer recover without a page reload.

### Messages

Add a constrained `kind` column:

- `user`: a normal member message;
- `system`: a moderation event generated by the server.

System messages may have no sender, do not count toward rate limits, and use dedicated rendering. Both kinds count toward the 36-message room limit.

### Attachment cleanup queue

Each queued item records the attachment ID, bucket, storage path, creation time, attempt count, last error, and completion time. A unique constraint prevents the same storage path from being queued twice.

## Sending and Rate-Limit Flow

1. The client validates text and any selected file.
2. If present, the client uploads the file and creates attachment metadata.
3. The client calls `send_chat_message` with the member ID, login code, room, content, and optional attachment ID.
4. The RPC locks the member row and verifies the credentials and attachment ownership.
5. If `chat_muted_until` is still in the future, the RPC rejects the request with the mute expiry. It creates no additional system message.
6. Otherwise, the RPC counts that member's `user` messages across every room whose `created_at` falls within the preceding 60 seconds.
7. If the count is at least 18, the attempted message is not inserted. The RPC sets `chat_muted_until` to exactly five minutes from the database clock and inserts one `system` message in the attempted room with this text: `Thằng SV <Tên> đã bị khóa mõm 5 phút`.
8. If the count is below 18, the RPC inserts the user message normally.
9. The RPC keeps the newest 36 messages in the affected room and deletes every older message in that room.
10. Attachments from pruned messages, or from a file uploaded for a rejected send, are added to the cleanup queue.

The attempted nineteenth message is rejected and therefore never appears in history. Repeated attempts during the five-minute mute do not extend the mute and do not create more system messages.

Database time is authoritative for all rolling-window and mute calculations. Network failures and client-side validation failures do not count as messages or violations.

## Retention and Attachment Cleanup

Retention is independent per room. After every accepted user message or generated system message, exactly the 36 newest messages remain when the room previously contained more than 36.

Deleting a message with an attachment detaches it and queues its storage object for deletion. The cleanup worker uses the Storage API rather than mutating the internal `storage` schema directly. Cleanup is idempotent: an already-missing object is treated as success, while transient failures preserve the queue item for retry.

The client immediately queues an uploaded attachment if its send is rejected. A periodic server-side sweep also finds unattached metadata older than a short safety interval and queues it, covering browser crashes between upload and send.

## Realtime and Notifications

Accepted user messages and moderation system messages are delivered through the existing `messages` Realtime subscription. System messages do not trigger the normal per-recipient message-notification fan-out, avoiding a second notification for the same moderation event.

The existing notification collection remains the unread source of truth. Every notification render updates:

- the header notification badge;
- chat and per-room badges; and
- `document.title`, using `(n) PingPing` when `n > 0` and `PingPing` when `n = 0`.

Counts greater than 99 may be displayed as `99+` in visual badges, but the document title uses the actual unread count.

The existing Notifications API behavior remains opt-in and only shows a system notification when the document is hidden. Closing the PingPing page ends notification delivery.

## User Interface

Opening or switching to a room loads all 36 retained messages in one query, orders them chronologically for display, and scrolls to the newest message. Incremental history pagination is unnecessary while retention remains capped at 36.

System messages are centered in the message stream without a member avatar or ordinary chat bubble. They have a distinct, subdued moderation style and accessible text.

While the current member is muted:

- the message input, attachment picker, and send button are disabled;
- the composer shows `Đã bị khóa mõm · còn mm:ss`;
- the effective name is `Súc vật <Tên>` in chat, the directory, and account UI; and
- a timer refreshes the countdown and restores the canonical display automatically at expiry.

Realtime member-state updates synchronize the mute across multiple open tabs and devices. The server still rejects sends if a stale client incorrectly leaves its composer enabled.

## Errors

The RPC returns distinguishable error codes for:

- invalid credentials;
- an active mute, including its expiry;
- a newly triggered rate limit, including its expiry;
- an attachment that is missing or belongs to another member; and
- invalid message or room data.

The controller maps these codes to Vietnamese status messages. It does not clear a user's draft after a rejected send. If an attachment was uploaded, it queues that object for cleanup and removes the local preview after the queue request succeeds.

Storage cleanup failures are recorded for retry and do not roll back a successfully accepted text message.

## Security

The RPC validates the member ID against the current login code because the project does not yet use Supabase Auth. It never trusts a client-provided timestamp, display name, sender name, mute expiry, or message kind.

Anonymous users cannot insert directly into `messages`, create system messages, change mute state, or read/write the private cleanup queue. The cleanup worker uses server-side credentials that are not exposed to the browser.

This improves the current chat boundary but does not make reusable login codes equivalent to full user authentication; replacing them with Supabase Auth remains separate work.

## Testing

### Database and schema

- The first 18 messages inside 60 seconds succeed; the nineteenth attempt is rejected.
- Messages across different rooms contribute to the same member limit.
- Concurrent sends cannot exceed 18 accepted messages.
- A violation creates exactly one five-minute mute and one system message in the attempted room.
- Attempts during an active mute do not extend it or add system messages.
- System messages do not count toward the rate limit.
- Each room retains its own newest 36 messages, including system messages.
- Pruned and rejected-send attachments are queued once.
- Direct anonymous message inserts and cleanup-queue access are denied.

### Client

- RPC result codes map to the correct Vietnamese state.
- Muted controls are disabled and the countdown reaches zero correctly.
- Effective names use the prefix only before the authoritative expiry.
- System messages render separately and escape their content.
- Opening a room requests up to 36 messages and renders every retained row.
- Unread totals produce `(n) PingPing` and reset to `PingPing` at zero.
- Realtime events update the mute state, message stream, unread badges, and title without duplicate appends.

### End-to-end verification

- Run the complete Node and frontend contract suite.
- Apply the migration and exercise the boundary cases against Supabase.
- Verify concurrent sends, room pruning, and Storage deletion.
- Verify desktop and mobile chat states in a browser.
- Run Supabase security and performance advisors and review new findings.

## Non-Goals

- Web Push, service workers, or notifications after closing the page;
- configurable rate-limit thresholds or mute durations;
- moderator controls for manually muting members;
- permanent changes to member names; and
- preserving chat history beyond the newest 36 messages per room.
